# 小航·郑州航院校园信息助手 —— 系统设计文档
## 一、系统概述
### 1.1 系统简介
本系统为轻量化校园智能问答助手，基于Python+Streamlit搭建前端交互页面，接入硅基流动DeepSeek大模型API，依托本地Markdown校园知识库，实现分身份差异化问答、多轮对话记忆、快捷提问、对话导出、异常容错等功能，面向郑航新生、在校生、教职工三类用户提供校园办事咨询服务。
### 1.2 设计目标
1. 轻量化部署，无需数据库、登录系统，打开网页即可使用；
2. 按用户身份分流Prompt，实现三种差异化回答语气与话术视角；
3. 严格约束大模型输出，杜绝信息幻觉、篡改校内电话/政策；
4. 完善异常处理机制，解决接口超时、网络故障、超长文本报错问题；
5. 提供对话导出、历史回溯、离线应急电话等配套辅助功能。
### 1.3 技术栈
| 分层 | 技术组件 | 用途 |
| ---- | ---- | ---- |
| 前端交互层 | Streamlit | 网页UI渲染、会话状态管理、按钮/输入框/标签页组件 |
| 业务逻辑层 | Python | 文本预处理、身份分流、对话上下文拼接、API请求封装 |
| 提示词管理层 | prompts.py | 人设定义、别名映射、防幻觉强制规则、知识库加载 |
| 大模型服务层 | 硅基流动 DeepSeek-V3.2 API | 云端大模型推理、对话生成 |
| 知识库存储层 | Markdown(.md) | 静态校园业务资料，离线可编辑更新 |
| 版本管理 | Git + GitHub | 代码迭代、云端仓库托管 |

## 二、总体架构设计
### 2.1 分层架构（三层架构）
1. **展示层（Streamlit UI）**
    负责页面渲染：标题、身份下拉框、4类快捷提问标签页、文本输入框、功能按钮、回答展示、提问历史、离线电话黄页；通过`st.session_state`临时存储会话数据。
2. **业务逻辑层（app.py 核心）**
    - 会话初始化：用户身份、对话上下文、提问历史持久化；
    - 文本预处理：同义词别名统一替换；
    - 身份读取与Prompt拼接；
    - API请求封装、超时重试、异常捕获；
    - 对话保存、导出文件、历史回填逻辑；
3. **数据与提示词层（prompts.py + data知识库）**
    - 读取全部md知识库文本；
    - 定义三类身份人设、别名转换规则、防幻觉强制约束；
    - 组装完整system系统提示词返回主程序。

### 2.2 整体业务流程图
用户访问页面 → 初始化session会话 + 加载全部本地知识库
→ 用户选择身份（页面即时刷新，身份存入会话）
→ 两种提问渠道：快捷按钮一键填充 / 手动输入文本
→ 文本别名预处理，统一替换“郑航/ZUA/航院”等简称
→ 读取当前用户身份，拼接【系统Prompt + 知识库 + 历史对话 + 当前提问】完整消息体
→ 调用硅基流动API，内置2次超时重试、网络异常捕获
→ 校验接口返回数据，渲染对应身份风格回答，展示字数&耗时
→ 将本轮问答存入上下文记忆，支持连续追问
→ 配套辅助操作：清空对话、导出TXT记录、点击历史问题二次提问
→ 接口故障时可查看底部离线校园应急电话黄页

## 三、模块详细设计
### 3.1 身份分流模块
1. 组件：`st.selectbox`下拉选择框，选项：新生/在校生/教师
2. 存储：身份存入`st.session_state.user_role`，页面切换自动刷新
3. 逻辑：每次发起AI请求时读取会话身份，传入`get_system_prompt()`生成专属人设提示词
4. 约束：教师身份强制禁止使用“同学”称呼，新生统一称呼学弟学妹，在校生标准称呼同学。

### 3.2 快捷提问标签模块
1. 标签分类：3个Tab：新生指南、办事流程、应急防骗；
2. 每个Tab内2列布局快捷按钮，点击自动填充问题至输入框并刷新页面；
3. 按钮Key唯一化，避免Streamlit组件冲突。

### 3.3 文本预处理模块
- 别名词典：内置INPUT_ALIAS，自动替换院校、校园设施简称；
- 超长文本校验：字符大于600弹出提示，建议拆分提问，降低接口超时概率。

### 3.4 多轮对话上下文模块
- 存储容器：`st.session_state["messages"]`列表，按顺序存储user/assistant对话；
- 新对话功能：一键清空messages列表，重置对话记忆；
- 拼接规则：每次请求时，系统Prompt置于列表首位，后拼接全部历史对话，最后追加当前用户提问。

### 3.5 提问历史模块
- 存储容器：`st.session_state.question_history`数组，自动去重存入提问；
- 功能：倒序展示全部历史提问，点击按钮回填至输入框；支持一键清空全部历史。

### 3.6 对话导出模块
- 格式：UTF-8纯文本`.txt`，自动生成带当日日期的文件名；
- 内容区分：标注【用户提问】【AI回答】，分割线分隔每条问答；
- 组件：`st.download_button`前端一键下载，无需后端文件存储。

### 3.7 AI接口请求与异常容错模块
1. 请求参数固定配置：model、max_tokens=600、temperature=0.01；
2. 重试机制：超时自动重试2次，超过上限提示网络问题；
3. 异常分类捕获：
    - Timeout超时：提示拆分短问题/切换热点；
    - ConnectionError连接失败：提示无法访问接口；
    - 401密钥错误、403无权限、5xx服务器故障、404地址错误，分别给出对应提示；
4. 返回校验：判断choices字段存在性，空回答给出报错弹窗。

### 3.8 离线电话黄页模块
页面底部静态渲染Markdown表格，存储校内各部门24小时/办公电话，接口完全失效时作为离线兜底查询渠道，无需调用AI。

### 3.9 Prompt管理模块（prompts.py）
1. 知识库加载函数：遍历data文件夹读取全部.md文件，拼接完整参考资料；
2. ROLE_PROMPTS字典：存储三类身份语气、称呼强制规则；
3. 别名文本规则：给大模型的同义词转换约束；
4. 硬性防幻觉规则：10条强制约束，规范信息来源、数字号码、反诈提醒、输出格式、来源标注。

## 四、数据设计
### 4.1 内存会话数据（临时数据，页面关闭自动销毁）
| 会话键名 | 数据类型 | 作用 |
| ---- | ---- | ---- |
| user_role | str | 存储当前选中用户身份：新生/在校生/教师 |
| user_q | str | 快捷按钮回填临时问题缓存 |
| question_history | list[str] | 存储所有历史提问文本 |
| messages | list[dict] | 多轮对话上下文，结构：{"role":"user/assistant","content":"文本"} |

### 4.2 静态知识库文件（data/*.md）
- 存储格式：Markdown纯文本；
- 内容规范：按业务分类，完整记录办理流程、材料、地点、官方电话、收费标准；
- 读取规则：启动时一次性全量加载，全部拼接传入系统Prompt作为参考依据。

### 4.3 导出文件格式
后缀`.txt`，纯文本，固定头部标识，问答分段分隔，便于本地存档、打印。

## 五、接口设计（硅基流动API）
### 5.1 基础请求信息
- 请求地址：`https://api.siliconflow.cn/v1/chat/completions`
- 请求方式：POST
- 请求头：Authorization鉴权密钥 + Content-Type:json
### 5.2 请求体参数说明
```json
{
    "model": "deepseek-ai/DeepSeek-V3.2",
    "max_tokens": 600,
    "temperature": 0.01,
    "messages": [
        {"role":"system","content":"身份+知识库+强制规则"},
        {"role":"user","content":"历史提问1"},
        {"role":"assistant","content":"AI回答1"},
        {"role":"user","content":"当前提问"}
    ]
}
```
- temperature=0.01：降低模型自由发挥，严格遵循知识库原文，减少编造；
- max_tokens：限制单次回答最大长度，防止超长输出卡顿。
### 5.3 正常返回结构
返回JSON，`choices[0].message.content`为AI生成的回答文本。

## 六、异常与容错设计
1. **知识库缺失异常**
    data文件夹无md文件时，页面顶部黄色警告提示补齐知识库；
2. **API网络超时**
    最多2次自动重试，重试失败给出网络优化建议；
3. **接口鉴权/权限异常**
    区分401、403、404、5xx状态码，精准定位密钥、地址、服务器故障；
4. **AI返回空内容**
    弹窗提示缩短提问长度，拆分内容后重试；
5. **超长文本输入**
    前置弹窗预警，提前规避接口截断、超时问题；
6. **无网络兜底**
    页面底部永久展示离线校园电话黄页，无AI服务时可手动查阅官方联系方式。

## 七、安全与约束设计
1. **信息防篡改约束**
    Prompt强制要求校内电话、收费、部门名称一字不差照搬知识库，禁止修改数字；
2. **反诈强制约束**
    缴费、学费、转账相关问题，回答结尾自动添加防诈骗提醒；
3. **隐私访问限制**
    拒绝查询学生成绩、一卡通余额、个人教务隐私数据；
4. **违规内容拦截**
    拒绝代写论文、作弊、暴力、校外无关咨询；
5. **幻觉防控**
    知识库无匹配内容时统一引导拨打学校总机，禁止编造政策、电话、时间。

## 八、部署与运行设计
1. 本地运行：安装requirements依赖，终端执行`streamlit run app.py`，访问localhost:8501；
2. 版本托管：Git管理代码，推送至GitHub远程仓库保存迭代版本；
3. 轻量化优势：无需MySQL/Redis数据库、无需后端服务、无需用户登录系统，仅Python环境即可运行；
4. 扩展方案：支持迭代新增知识库文档、拓展提问分类标签，适配更多校园业务场景。

## 九、系统取舍说明
1. 不设计账号登录模块
    设计理由：轻量化工具，无需持久存储用户对话，关闭页面自动清空数据，简化部署；
2. 不接入本地向量库/本地大模型
    设计理由：适配学生普通PC硬件，依托云端API降低本地算力要求；
3. 限制单次提问文本长度上限600字
    设计理由：接口单次token存在限制，过长文本易触发超时、截断回答，拆分提问可提升交互稳定性。
