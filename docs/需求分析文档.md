# 【小航·郑州航院校园信息助手】需求分析文档（初稿）
## 一、项目概述
1. **项目名称**：小航·郑州航院校园信息助手
2. **项目目标**
    基于本地校园知识库与大语言模型，搭建面向郑航三类人群的轻量化校园问答网页工具。依据用户身份差异化输出对应语气、视角的答复，提供快捷提问、多轮对话记忆、对话导出、提问历史回溯等功能；严格约束AI输出，杜绝编造信息、诈骗话术，实现校园办事、生活、防骗信息一站式自助查询。
3. **技术栈**
    前端页面：Streamlit
    大模型接口：硅基流动DeepSeek-V3.2
    本地知识库：Markdown文档
    版本管理：Git + GitHub
    开发语言：Python
4. **小组成员**：个人独立开发（梁6-yl）

## 二、用户分析
### 2.1 用户类型A：新生
- 特点：不熟悉校园全部流程，对缴费、报到、宿舍、军训等基础业务完全陌生，容易被电信诈骗，需要通俗、细致的科普讲解。
- 高频需求：报到流程、学费缴纳、宿舍规格、军训物资、校园卡使用。
- 优先级：最高（核心服务人群）
### 2.2 用户类型B：在校生
- 特点：熟悉基础校园生活，重点关注教务、转专业、图书馆、报销、调课等进阶业务，追求简洁高效的办事材料、地点、时间信息。
- 高频需求：在读证明办理、转专业渠道、图书馆开放时间、差旅报销流程。
- 优先级：高
### 2.3 用户类型C：教师
- 特点：行政办公视角，需要规范、书面化政策说明，关注审批流程、学生管理相关制度。
- 高频需求：学生缓缴学费流程、学生求助处置、校园安全管理规范。
- 优先级：中

## 三、功能需求
### 3.1 F01 身份切换功能（高优先级）
- 描述：下拉框提供新生/在校生/教师三种身份，切换后页面实时刷新，全局保存身份状态；AI读取身份生成对应人设回答，页面实时展示当前生效身份用于调试。
### 3.2 F02 快捷分类提问（高优先级）
- 描述：顶部标签栏分为「新生指南、办事流程、应急防骗」3个模块，每个模块内置快捷提问按钮，点击自动填充问题至输入框并触发AI查询；移除原交通出行分类模块。
### 3.3 F03 自定义文本提问（高优先级）
- 描述：文本输入框支持手动输入校园相关问题，自动替换学校别名（郑航/ZUA/航院）统一标准名称；超长文本弹出提示，降低接口超时概率。
### 3.4 F04 多轮对话上下文记忆（高优先级）
- 描述：自动保存用户与AI完整对话记录，支持上下文连贯问答；提供「开启新对话」一键清空全部历史上下文。
### 3.5 F05 提问历史回溯（中优先级）
- 描述：页面底部展示全部历史提问，点击历史问题可快速回填至输入框；支持一键清空全部提问历史。
### 3.6 F06 对话记录导出（中优先级）
- 描述：一键导出全部对话内容为本地TXT文件，自动携带当日日期命名，区分用户提问与AI回答。
### 3.7 F07 AI接口异常容错（高优先级）
- 描述：接口超时自动重试2次，区分超时、连接失败、密钥错误、服务器故障等异常并给出对应提示；限制单次最大输出字符，统一响应耗时、回答字数展示。
### 3.8 F09 离线校园电话黄页（低优先级）
- 描述：页面底部内置各部门官方联系电话，AI接口故障时可离线查阅，覆盖保卫处、校医院、招生办、后勤报修等场景。

## 四、应用流程
### 4.1 主流程描述
1. 用户打开网页，系统自动初始化会话、加载本地全部Markdown知识库；
2. 用户下拉选择自身身份，页面刷新并记录身份；
3. 两种提问方式：点击快捷提问按钮 / 手动输入自定义问题；
4. 系统替换文本别名，存入提问历史；
5. 读取当前身份，拼接专属身份提示词+知识库+历史上下文，调用硅基流动API；
6. AI返回对应身份口吻的规范回答，页面展示字数、耗时；
7. 保存本轮问答至上下文，支持继续追问；
8. 用户可清空对话、导出记录、点击历史问题二次查询；
9. 接口异常时展示报错提示，底部可查看离线应急电话。
### 4.2 流程图
页面访问 → 初始化会话&加载知识库 → 用户选择身份 → 输入/点击快捷问题 → 别名文本预处理 → 读取身份拼接系统提示词 → 调用AI接口（含重试容错）→ 渲染差异化回答 → 保存对话上下文 / 支持清空、导出、历史回溯

## 五、数据设计
### 5.1 数据格式选型
本地静态知识库：`.md` Markdown纯文本格式，便于修改、阅读、拆分业务文档；
对话导出文件：`.txt` 纯文本，无格式兼容门槛；
会话临时数据：Streamlit session_state内存存储，页面关闭自动清空。
### 5.2 数据文件清单
| 文件/文件夹 | 用途 |
| ---- | ---- |
| data/文件夹 | 存放全部校园业务知识库md文档 |
| app.py | 页面、交互、接口调用主程序 |
| prompts.py | 身份人设、别名规则、防幻觉提示词管理 |
| requirements.txt | 项目依赖库清单 |
| README.md | 项目部署说明 |
| .gitignore | 忽略虚拟环境、缓存、日志文件 |
### 5.3 写作规范
1. Markdown知识库：按业务分类拆分文档，统一标注部门、电话、办理时间、材料；数字、号码原文固定不修改；
2. 对话导出文本：固定头部说明，每条问答分段分隔，清晰区分用户与AI内容；
3. 代码注释：关键功能、会话状态、接口参数添加注释，便于迭代维护。

## 六、Prompt设计
### 6.1 身份分流策略
1. 新生人设：热心大二学长，口语温柔科普，称呼学弟学妹，缴费场景强制反诈提醒；
2. 在校生人设：办事熟手，简洁干货，优先地点、材料、时间，统一称呼同学；
3. 教师人设：正式书面行政口吻，面向管理审批场景，全文禁用“同学”称呼，替换为教职工/老师；
### 6.2 别名词典
内置文本替换规则：ZUA/郑航/航院统一替换为郑州航空工业管理学院；饭卡=校园一卡通、在读证明=学籍在校证明等，AI自动识别同义词汇。
### 6.3 防幻觉硬性规则
1. 仅依托本地知识库作答，无匹配内容统一引导拨打学校总机；
2. 所有电话、收费、时间、部门名称一字不差复刻知识库，禁止篡改数字；
3. 严禁编造教职工私人电话、未公布政策；
4. 学费、转账类问题结尾强制添加反诈提醒；
5. 回答末尾必须标注对应知识库文档来源，禁止截断、删减步骤；
6. 逐字校对输出，杜绝错别字、语病、重复叠词。

## 七、用户边界声明
### 能聊什么
郑州航空工业管理学院校内全部业务：新生报到、缴费住宿、教务办理、图书馆、后勤报修、校园反诈、应急求助、各部门办公流程；校内交通相关知识库内容已移除，不再支持交通类问答。
### 不能聊什么
1. 非本校无关问题、校外院校政策；
2. 学生隐私（成绩、一卡通余额、个人教务信息）；
3. 代写论文、作业、作弊类请求；
4. 领导私人联系方式、未公开校内内部文件；
5. 心理危机仅提供官方热线，不做专业心理诊疗；
6. 交通出行类相关问题无知识库支撑，无法作答。
### 数据更新日期
知识库内容基于2026年郑州航院公开办事指南，后期可新增/修改md文档更新数据。

## 八、不做的事（功能删减说明）
1. 不做交通出行分类模块
    理由：需求精简，移除交通相关快捷提问入口，减少知识库维护成本，聚焦新生、教务、安全三大核心场景。
2. 不做用户登录、账号系统
    理由：轻量化单页面工具，无需持久存储用户数据，会话关闭自动清空记录，降低部署复杂度。
3. 不做向量检索/本地大模型部署
    理由：依托云端硅基流动大模型，降低本地硬件要求，开箱即用，适配学生开发环境。
4. 不支持长文本一次性完整解析（600字以上弹窗提醒拆分）
    理由：接口单次token限制，过长文本易触发超时、截断回答，拆分提问提升稳定性。
